# 13期计划：真实链路回归与首响应止损

## 根因

- 2026-07-21 线上真实请求依次命中额度耗尽、限流和 502 的四个 Provider，单条消息累计约 80 秒后才落到可用模型。
- 插件只扫描消息顶层组件，未递归识别引用消息 `Reply.chain` 内的图片；因此可能先选择纯文本模型，再被 AstrBot 核心切回正在冷却的多模态默认模型。
- 之前的发布验证只证明配置、冷却和候选注入状态，没有逐个真实请求所有 Provider，无法提前暴露上游故障。

## 本期目标

- OpenAI-compatible Provider 普通调用和流式首响应默认最多等待 20 秒；超时立即按单模型错误冷却并继续 fallback。
- 递归识别引用消息内的图片与语音，在路由和安全候选生成阶段完成模态过滤。
- 发布后对实时 fallback 链每个 Provider 各发一次最小真实请求，并通过 WebChat 跑默认模型到可用 fallback 的端到端请求。

## 验证

- 单元测试覆盖普通调用超时、流式首 chunk 超时、嵌套引用图片/语音和循环引用保护。
- 容器内运行完整测试与 `compileall`，校验 `_conf_schema.json`。
- 逐个记录全部实时 Provider 的成功/失败、错误类型与耗时，不输出 API Key。
- 端到端请求必须在失败模型冷却后选中真实可用模型并返回内容，决策日志顺序与 `cmd_config.json` 一致。

## 状态

- v0.11.2 已完成并同步实时 Docker 环境。
- 容器内 55 项单元/集成测试通过，源码 `compileall`、配置 JSON 和 `git diff --check` 通过。
- 部署后逐个真实请求当前 7 项链路：`gpt-5.6-sol` 返回 429；两个 Grok 4.5 返回 429；`gpt-5.4` 返回 502；Mimo、opencode Flash、自有 DeepSeek 分别在 4.08、2.52、0.78 秒返回 `PONG`。
- 实时配置确认首响应上限为 20 秒、单模型尝试次数为 1；真实 WebChat 请求按冷却状态从默认模型切到 Mimo，9.59 秒出首 Token，10.25 秒完整返回 `ROUTER_OK`。
- 测试过程中创建的两个 WebChat 会话已通过 Dashboard API 删除。
